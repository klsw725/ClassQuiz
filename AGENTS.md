> 코드를 읽으면 알 수 있는 것은 여기 적지 않는다.
> 여기에는 지뢰만 있다.

## 툴링 지뢰
- **백엔드 테스트는 plain `pytest`가 아니다**. CI는 secret `.env`를 만든 뒤 `./run_tests.sh a`를 실행하고, 이 스크립트는 `podman compose`, `/tmp/storage`, Alembic, `--asyncio-mode=strict`까지 묶어 수행한다. 백엔드 검증은 먼저 필수 env를 준비하고 이 경로를 따르며, 실패한 일부 테스트만 임의로 단독 실행했다고 완료 처리하지 말 것.
- **Python 버전 기준이 서로 어긋나 있다**. 개발 문서/CI는 3.10 계열을 말하지만 `Pipfile`/락/Docker 계열은 더 최신 런타임을 요구한다. 버전 관련 실패를 한 파일만 고쳐 “정리”하지 말고 Pipenv 락, Docker, CI, 문서를 함께 맞출 것.
- **프론트 CI는 전체 로컬 검증이 아니다**. GitHub Actions는 `pnpm run lint-without-format-checking`만 실행하고, `pnpm run lint`의 Prettier 검사와 `pnpm run check`의 `svelte-check`는 빠져 있다. 프론트 변경 후 CI 통과만 믿지 말고 `frontend/`에서 pnpm 기반 lint/check/build를 직접 돌릴 것.

## 아키텍처 지뢰
- **백엔드 import는 순수하지 않다**. `classquiz.config`는 `settings()`를 캐시하고 import 시 Redis/ARQ/storage/Meilisearch 객체와 URL 정규식을 만든다. 환경변수나 `.env`는 첫 import 전에 확정하고, 테스트 중 env를 바꿔 재import로 반영된다고 가정하지 말 것.
- **Alembic은 `alembic.ini` URL보다 앱 설정을 따른다**. migration env가 앱 모델과 `settings.db_url`을 import하므로 DB URL만 있어도 되는 표준 Alembic처럼 동작하지 않는다. 마이그레이션 실행 환경에는 DB 외에도 앱 필수 설정(`SECRET_KEY`, mail, storage 등)을 함께 제공할 것.
- **API worker 수 1은 의도된 제약이다**. Socket.IO 서버가 Redis/message-queue client manager 없이 프로세스 로컬 room/session 상태를 쓴다. `MAX_WORKERS=1`을 성능 튜닝으로 올리지 말고, 올리려면 Socket.IO 멀티프로세스 설계를 먼저 바꿀 것.
- **Socket.IO 루트 mount는 마지막이어야 한다**. `/`에 `ASGIApp(sio)`를 mount하므로 앞에 두면 `/api/v1/...` 라우트까지 가로챌 수 있다. 새 FastAPI 라우터는 루트 mount보다 위에 등록할 것.

## 설정/런타임 지뢰
- **Redis 게임 상태는 alias와 단계가 나뉜다**. `game:{pin}`, `game_pin:{user_id}:{quiz_id}`, `game_in_lobby:{user_id}`가 먼저 생기고 `game_session:{pin}`은 Socket.IO admin 등록 후에야 생긴다. live/remote/컨트롤러 흐름을 고칠 때 pin과 quiz id를 단일 키로 합치거나 세션 존재 전제를 만들지 말 것.
- **CKEditor `licenseKey: 'GPL'`은 청소 대상이 아니다**. 현재 CKEditor 5 설정은 라이선스 키 필수 요구사항을 GPL 키로 만족시키는 형태다. 에디터 코드를 정리할 때 이 값을 제거하거나 상용 키로 바꾸지 말고 라이선스 정책을 먼저 확인할 것.

### 2. 단순함 우선

**문제를 해결하는 최소한의 코드만 작성하라. 추측성 구현 금지.**

- 요청되지 않은 기능은 추가하지 마라.

- 요청되지 않은 "유연성"이나 "설정 가능성"을 추가하지 마라.
- 발생할 수 없는 시나리오를 위한 에러 처리를 만들지 마라.
- 200줄을 썼는데 50줄로 가능하다면 다시 작성하라.

### 3. 외과적 변경

**정말 필요한 부분만 수정하라. 자신의 변경으로 생긴 것만 정리하라.**

- 주변 코드, 주석, 포맷을 "개선"하지 마라.
- 고장 나지 않은 것을 리팩토링하지 마라.

- 관련 없는 데드 코드가 보이면 언급만 하고 삭제하지 마라.
- 본인의 변경으로 사용되지 않게 된 import/변수/함수는 제거하라.
- 기존에 존재하던 데드 코드는 요청받지 않았다면 제거하지 마라.
